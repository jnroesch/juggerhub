# Implementation Plan: Events

**Branch**: `006-events` | **Date**: 2026-07-04 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `/specs/006-events/spec.md`

## Summary

Turn the minimal seeded `Event` (its own comment already calls it "the foundation for a later events feature") into a **real, ownable event**. Any signed-in user creates one through a guided, onboarding-style **wizard** (type & name → dates → location → who-can-join + limit → fee → review → publish) and becomes its first **admin**. Each event lives at `/events/{id}` (UUIDv7 URL, no slug) as a **public** page anyone can read: name/type/description, start/end, an in-person **address (incl. country)** or a **virtual link**, an information-only **fee**, the latest **news**, three participant groups — **joined / awaiting approval / waiting list** — and a free-form **contacts** list. Signed-in users **sign up** (as themselves for individuals-only events, or by entering a **team they administer** for teams-only): **free + open spot → joined**, **paid + open spot → awaiting approval** (a provisional hold; pay out-of-band, admin confirms), **full → waiting list** (never charged until promoted). **Occupied spots = joined + awaiting-approval**; waitlist never counts; capacity is enforced atomically so concurrent last-spot sign-ups can't exceed the limit. **Nothing auto-fills** — every admission is an explicit admin **approve** or **promote**. Admins also **remove** participants, **post news**, add/update/remove **contacts**, **invite co-admins** (share link or targeted email, all admins equal under a **last-admin guard**), **edit** (mode locked once sign-ups exist; limit can't drop below occupied), and **cancel** (irreversible; page stays up marked cancelled; joined + waiting notified by **email**).

The approach adds a new domain slice under `Services/Events/` following the established **thin-controller / DI-service / EF-Core-directly / Mapster-DTO / `BaseEntity`(UUIDv7)** conventions, reusing almost everything the 005 team slice built: the **capacity/last-admin transaction pattern** (`SELECT … FOR UPDATE` on the event row, mirroring `TeamService.MutateMembershipAsync`), the **invitation model** (`InvitationKind`/`InvitationStatus` enums, raw capability token, partial-unique active-link/pending-targeted indexes, anonymous preview + authed accept), the **email pipeline** (`IEmailSender`/`IEmailTemplateService`, Mailpit/Resend), and the **`PagedResult<T>`** contract. The existing `Event` entity is **extended in place** (adding type/description/schedule/location/mode/limit/fee/status), and the existing `EventParticipation` (profile-centric **activity/attendance**, seeded) is **left untouched** — the new **`EventSignup`** entity is the live registration workflow and is deliberately distinct. Five new entities — **`EventSignup`**, **`EventAdmin`**, **`EventAdminInvitation`**, **`EventContact`**, **`EventNewsPost`** — plus five new enums and one migration (`AddEvents`). Authorization is strictly server-side: the event **page and its lists are public** (a prospective participant must decide before signing in), **sign-up requires auth**, and every **admin action** is gated on `EventAdmin` membership; the last admin can never be removed. The frontend adds an `EventService`, the stepped create wizard, the public event page with an in-page admin toolkit, manage/contacts/admins screens behind `authGuard`, and a full-screen anonymous co-admin accept page — all styled from `DESIGN.md`, validated desktop + mobile.

## Technical Context

**Language/Version**: Backend — C# 13 on .NET 10 (ASP.NET Core, EF Core 10, ASP.NET Core Identity). Frontend — TypeScript on Angular (standalone components, signals) in an Nx workspace.

**Primary Dependencies**:
- Backend (existing, reused): `Microsoft.AspNetCore.Identity.EntityFrameworkCore`, `Microsoft.AspNetCore.Authentication.JwtBearer`, `Npgsql.EntityFrameworkCore.PostgreSQL`, `Mapster`, `Asp.Versioning.Mvc`, the existing email pipeline (`IEmailSender`, `IEmailTemplateService`, `EmailOptions.FrontendBaseUrl`). **No new NuGet packages** — capability tokens use `RandomNumberGenerator`; capacity + last-admin invariants use EF transactions with a row lock.
- Frontend: `@angular/*` (router, reactive forms, signals), RxJS, Tailwind; `jest` (unit), `@playwright/test` (e2e). **No new runtime dependency.**

**Storage**: PostgreSQL 18. **Extended** table: `Events` (+ type/customLabel, description, startsAt/endsAt, locationKind + address parts + virtualLink, participantMode, participationLimit, fee columns, status/cancelledDate; the single `Date DateOnly` becomes `StartsAt`/`EndsAt DateTime`). **New** tables: `EventSignups`, `EventAdmins`, `EventAdminInvitations`, `EventContacts`, `EventNewsPosts`. One EF migration `AddEvents` (auto-applies on startup). Existing seeded events are regenerated by `DevDataSeeder` (Dev/local only; no Prod event data exists yet).

**Testing**: Backend — xUnit + `WebApplicationFactory<Program>` + `Testcontainers.PostgreSql` (extend the existing integration project): create validation (name, end≥start, country for in-person, link for virtual, positive limit, paid requires recipient+IBAN); **capacity atomicity** (concurrent last-spot sign-ups → exactly one joined, surplus waitlisted); sign-up routing (free→joined, paid→awaiting, full→waitlist) + mode mismatch + team-admin-only entry + duplicate guard; **approve / promote (capacity-guarded) / remove**, no auto-promotion; **edit guards** (mode locked once sign-ups exist, limit not below occupied); **event last-admin guard** (incl. concurrent race); admin-invite link/targeted/expiry/revoke + accept grants co-admin; contacts validation (≥1 method) + admin-only; news admin-only; **cancel** blocks sign-ups/approve/promote + sends emails; public page anonymous (no auth); pagination on every list; non-admin refusals across the board. Frontend — Jest unit (`EventService`, wizard step validators, capacity/role gating) + Playwright `events.spec.ts` (create wizard → view as visitor → sign up as 2nd user → admin approve/promote/remove → post news + contact → invite co-admin → cancel), desktop + mobile.

**Target Platform**: Linux containers via Docker (`docker-compose`). Product UI targets desktop + mobile (responsive web).

**Project Type**: Web application — existing sibling `backend/` (.NET) and `frontend/` (Nx/Angular) trees.

**Performance Goals**: No throughput targets. All list reads use projections + `AsNoTracking` + `PagedResult<T>`; participant/news/contacts/invitation/user-search queries hit `EventId`/composite indexes; token lookups hit unique indexes. Capacity checks and the last-admin guard are short `Serializable` transactions with a `SELECT … FOR UPDATE` on the event row (rare, brief contention).

**Constraints**: Security-first / OWASP / never-trust-the-client — **all** authorization server-side. The event page + its lists are **intentionally public** (public event fields only; no admin-only data such as pending-invite tokens or user emails leaks); **sign-up** requires auth and a mode/authority match; every **mutation** is gated on `EventAdmin` membership (or, for withdraw, the participant / their team's admin). Capacity and the event last-admin invariant are enforced atomically (row lock). Capability tokens are unguessable + expiring + revocable. Environment parity — email reuses Mailpit/Resend (one or two new templates, **no new infra/secrets**). `.ps1`-only scripts; Docker-only workflow; responsive UI at multiple viewports; `.html`/`.css`/`.ts` kept separate; enums serialize as names via the existing global `JsonStringEnumConverter`.

**Scale/Scope**: ~25 endpoints across events / sign-ups / participant-admin / news / contacts / admins / admin-invitations. One extended entity + five new entities + five new enums; one migration. Frontend: `EventService` + the stepped create wizard (5 steps + review), the public event page with in-page admin toolkit, manage-participants, contacts, admins/invitations, and a full-screen co-admin accept page. Extends `DevDataSeeder`. Sized so a later events-index/discovery, calendar export, and signup→activity bridge consume these foundations without reshaping them.

| — | Secret & Configuration Management | No new secrets. Optional `EventOptions` (invite-link TTL days, name/description/limit bounds, news/contact caps) are plain config with safe defaults; no Key Vault. Fee IBAN/recipient are event content the organiser enters, not app secrets. | ✅ |

**Result**: PASS — no violations; Complexity Tracking left empty. The **polymorphic sign-up subject** (a signup carries a `UserId` XOR a `TeamId`) is a deliberate modeling choice (research §3), bounded by a DB `CHECK` (exactly one set) + two partial-unique indexes — a normal domain constraint, not a constitution deviation. Exposing the event's fee **recipient/IBAN publicly** is intended payment-instruction content the organiser chooses to publish (research §6), not a leak of app-held secrets.

**Structure Decision**: Web application extending the existing `backend/` and `frontend/` trees. Backend stays organized by technical type with new logic grouped under `Services/Events/` (joining the existing `EventActivityService` there and mirroring `Services/Teams/`), and the event emails under `Services/Email/` next to `AuthEmailService`/`TeamEmailService`. The surface splits by resource: event-scoped public/admin routes on `EventsController`, token-addressed co-admin routes on `EventInvitationsController`. The **public event page** lives **inside** the shell **without** `authGuard` (like the dashboard) so anonymous visitors and signed-in users share one page (admin toolkit + sign-up gated server-side and shown conditionally); the create wizard and admin management screens sit inside the shell **behind** `authGuard`; the co-admin accept page is a full-screen **anonymous** route **outside** the shell (like `/join/:slug/:token`), requiring auth only to act. No new project or library is introduced.
